## 1. 这轮审计在回答什么

上一轮 `archive/tsuyama/45_phase_gate跨对象敏感性复核.md` 里，
我发现部分 `404 / 488` 主路线 exosome case 的独立 rerun 无法复现当前全量主库。

但那一轮 rerun 用的是：

- `n_events = 1000`
- `lockin_time_constant_s = 1.0 ms`

而当前全量主库 `results/fine_full_range_biomimetic_exosome_10000e_meta.json` 记录的是：

- `n_events = 10000`
- `lockin_time_constant_s = 1.5 ms`

所以那一轮只能证明“值得进一步审计”，不能直接证明“主库 provenance 已失配”。

这次新审计的目的，就是用 **严格对齐的 replay** 来回答：

1. 如果完全按主库 `meta.json` 重放，当前代码还能不能复现当前全量主库？  
2. 如果不能，偏差到底只是当前默认 `tau = 1.0 ms` 带来的，还是已经存在真正的主库/代码不一致？

---

## 2. 审计设置

### 2.1 case 选择

沿用上一轮挑出的 `exosome_mainline_representative` 四个主路线代表 case：

1. `exosome_biomimetic_corona_nominal_80nm`, `404 nm`, `500 x 1800 nm`
2. `exosome_biomimetic_corona_nominal_140nm`, `404 nm`, `500 x 1700 nm`
3. `exosome_biomimetic_corona_nominal_80nm`, `488 nm`, `600 x 1600 nm`
4. `exosome_biomimetic_corona_nominal_140nm`, `488 nm`, `500 x 2000 nm`

### 2.2 replay profile

这次跑了两组 profile，统一用 `8 workers` 并行：

#### A. `stored_meta_exact`

直接读取全量主库 `meta.json` 的 `SimulationConfig`：

- `n_events = 10000`
- `lockin_time_constant_s = 1.5 ms`
- `score_mode = single`

它代表的问题是：

> 如果当前代码完全按主库记录的参数重放，能不能贴回当前主库？

#### B. `current_dashboard_replay_10000e`

基于当前 `dashboard/config.py::DEFAULT_SIM_CFG`，只做两件必要对齐：

- 把 `n_events` 补到 `10000`
- 把 `random_seed` 对齐到 `42`

此外为了能做单 case replay，把 `score_mode` 从 dashboard 默认 `joint` 暂时切到 `single`。
这是为了让单 case 重放能执行；它不会改变这里关心的原始 detection / gate 统计。

在这个 replay 口径下，和 `stored_meta_exact` 相比，真正剩下的配置差异只有一项：

- `lockin_time_constant_s`: `1.5 ms -> 1.0 ms`

## 3. 结果

### 3.1 汇总

`results/mainline_consistency_audit_report.json` 的结论非常明确：

- `stored_meta_exact`：
  - `gate_match_count = 1 / 4`
  - `blocker_match_count = 1 / 4`
- `current_dashboard_replay_10000e`：
  - `gate_match_count = 0 / 4`
  - `blocker_match_count = 0 / 4`

这说明：

1. 当前 dashboard 默认 `1.0 ms` 的 replay，确实和当前主库明显不一致。  
2. 但更重要的是，**即使完全按主库 `meta.json` 精确重放，当前代码也只能复现 4 个 case 里的 1 个。**

因此，问题已经不能再被解释成“只是默认值从 `1.5 ms` 改成了 `1.0 ms`”。

### 3.2 四个代表 case 的逐项对比

#### Case A  
`exosome_biomimetic_corona_nominal_140nm`, `404 nm`, `500 x 1700 nm`

- 当前主库：`detection_rate = 0.4907`, `stable_detection_rate = 0.4855`, `pass = True`
- `stored_meta_exact`：`detection_rate = 0.4720`, `stable_detection_rate = 0.4666`, `pass = True`
- `current_dashboard_replay_10000e`：`detection_rate = 0.2386`, `stable_detection_rate = 0.2353`, `pass = False`

这说明：

- 这个 case 在当前代码下 **可以被主库精确口径近似复现**
- `1.5 ms -> 1.0 ms` 会显著拉低它，但不是把它直接打到 `0`

#### Case B  
`exosome_biomimetic_corona_nominal_140nm`, `488 nm`, `500 x 2000 nm`

- 当前主库：`detection_rate = 0.6713`, `stable_detection_rate = 0.6554`, `pass = True`
- `stored_meta_exact`：`detection_rate = 0.0000`, `stable_detection_rate = 0.0000`, `pass = False`
- `current_dashboard_replay_10000e`：`detection_rate = 0.0000`, `stable_detection_rate = 0.0000`, `pass = False`

这是这轮最关键的证据。

按当前代码的物理口径：

- `NA_collection = 0.9`
- `W_min ≈ λ / NA_collection`
- 对 `488 nm`，`W_min ≈ 542 nm`

因此 `W = 500 nm` 本来就低于当前 `NA cutoff` 可收集阈值，
它不应该在当前核心里继续作为高检测率通过 case 存在。

换句话说：

- **这个主库 case 与当前代码主链已经明显不一致**
- 而且这不是小数值漂移，而是“当前代码下应接近失效、主库里却高分通过”的级别

#### Case C  
`exosome_biomimetic_corona_nominal_80nm`, `488 nm`, `600 x 1600 nm`

- 当前主库：`detection_rate = 0.3864`, `stable_detection_rate = 0.3817`, `pass = True`
- `stored_meta_exact`：`detection_rate = 0.1920`, `stable_detection_rate = 0.1840`, `pass = False`
- `current_dashboard_replay_10000e`：`detection_rate = 0.0918`, `stable_detection_rate = 0.0894`, `pass = False`

这个 case 不像 Case B 那样“物理上明显越过 NA cutoff 红线”，
但它仍然说明：

- **当前主库的 `488 / 600 / 80nm` 高分结论，也无法被当前代码按原 `meta` 口径完整复现**

#### Case D  
`exosome_biomimetic_corona_nominal_80nm`, `404 nm`, `500 x 1800 nm`

- 当前主库：`detection_rate = 0.2717`, `stable_detection_rate = 0.2691`, `pass = True`
- `stored_meta_exact`：`detection_rate = 0.0363`, `stable_detection_rate = 0.0336`, `pass = False`
- `current_dashboard_replay_10000e`：`detection_rate = 0.0209`, `stable_detection_rate = 0.0202`, `pass = False`

这个 case 表明：

- 不一致并不只集中在 `488` 或 `NA cutoff` 边界附近
- 对一些低对比度、低尺寸的 `404 nm` exosome case，当前主库与当前代码也已经出现明显偏离

## 4. 这轮审计后的判断

### 4.1 需要先撤回什么

上一轮我把“当前代码不复现主库”直接写得偏重了。  
现在要更准确地改成：

- 上一轮 `1000e + 1.0 ms` 的快跑结果，本身不足以直接证明 provenance 失配
- 但它的怀疑方向是对的
- 因为这轮 `meta.json` 精确重放之后，**确实证明了当前主库与当前代码并非整体一致**

### 4.2 现在能明确成立的结论

1. **Tsuyama gold 支线的 scoped fix 仍然成立。**  
   那部分问题主要是 readout / phase gate 语义，与这里的主库 provenance 问题不是同一件事。

2. **当前 `fine_full_range_biomimetic_exosome_10000e_*` 不能再被当成“完全对应当前代码”的标准主库。**  
   至少对部分 `404 / 488` exosome case，它已经和当前核心不一致。

3. **这种不一致不只是 `1.5 ms -> 1.0 ms` 的默认值切换。**  
   因为 `stored_meta_exact` 也无法复现多个 case。

4. **全量重算现在已经从“可选项”变成“主路线要继续可信推进时的必要项”。**

---

## 5. 现在是否需要全量重算

我的当前判断是：

### 5.1 如果只看 Tsuyama gold validation 支线

- **不需要**因为这次 scoped fix 立刻重算整套全量库  
- 因为那部分修正只改了验证支线的判据口径，没有改全局主线

### 5.2 如果要继续把当前 biomimetic exosome 全量库当成主结论来源

- **需要全量重算**

原因已经足够充分：

1. 当前主库里至少存在 `488 nm, 500 nm width` 这类与当前 `NA cutoff` 物理边界明显冲突的高分 case  
2. `stored_meta_exact` 精确重放也无法复现多个主路线代表 case  
3. 当前 dashboard 默认口径已经是 `1.0 ms`，而现存主库记录的是 `1.5 ms`

所以如果我们还想让：

- dashboard 默认结论
- 主路线 exosome 排序
- `recommended_default`
- 404 / 488 / 532 / 660 的横向比较

继续作为“当前代码的输出”被引用，
那现在就应该把全量库按当前代码重新生成一轮。

---

## 6. 建议顺序

我建议按下面顺序推进：

1. 先把当前 `fine_full_range_biomimetic_exosome_10000e_*` 明确视为“历史/过渡库”，不要再把它当完全 current 的标准真值  
2. 确认下一轮正式全量重算采用的 lock-in 默认：
   - 若坚持当前文档口径，应采用 `1.0 ms`
   - 若要保留与这份主库的连续性，则必须明确说明为什么继续用 `1.5 ms`
3. 按当前代码正式全量重算 biomimetic exosome 主库  
4. 全量重算完成后，再重新做：
   - 主路线排序/推荐分析
   - `phase_flip_fraction` 跨对象敏感性复核
   - Tsuyama support scope 与 exosome 主路线之间的证据边界说明

---

## 7. 当前结论一句话版本

**主路线现在已经不是“先继续沿用旧主库，后面再说”了；如果要让 exosome 全量结论继续代表当前代码，就需要正式重算全量库。**
