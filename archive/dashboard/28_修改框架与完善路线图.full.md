## 1. 当前阶段的单一结论

当前 dashboard 最合理的产品结构应改为：

- **前 6 页**：全部围绕标准 `fine_full_range_biomimetic_exosome_10000e_*` 结果库目标做分析、解释与筛选
- **最后 1 页**：保留为 `Single-Case Calculator`，用于自定义输入并单独计算

也就是说，dashboard 的主叙事不再是“coarse / live / fine 混合切换”，而是：

1. 先基于已经冻结的标准全量结果库看规律、看结论、看推荐
2. 只有在用户确实想看一个具体 case 的完整计算链时，才进入单案例计算页

因此，本轮改造**不是**继续补 physics；本轮改造的核心是：**把 dashboard 的展示逻辑彻底收口到标准全量结果库目标口径，并等待新库生成后直接接入。**

## 3. 现有 dashboard 的核心问题

### 3.1 数据源叙事不够单一

当前 dashboard 同时存在：

- `coarse_full_range_*`
- `fine_full_range_*`
- `live sweep`
- `single-case live`

这会带来三个问题：

- 用户不容易确定当前看到的结论到底基于哪一套数据
- 不同页面对“结果库分析”和“现场计算”的边界不够一致
- 代码里需要维护多套 `prefix / using_live_data / session_state` 分支

### 3.2 前 6 页的产品定位还不够统一

当前前 6 页虽然已经形成：

- `Principle Guide`
- `Mie Explorer`
- `Interference Explorer`
- `Noise & Detection Explorer`
- `Design Explorer`
- `Case Inspector`

但它们内部的逻辑并不完全一致：

- `Design Explorer` / `Case Inspector` 更像结果库分析页
- `Interference Explorer` / `Noise & Detection Explorer` 更像单案例机制页
- `Mie Explorer` 更像独立的本征散射计算页

这会让“主流程”在产品体验上看起来像是三种页面混合，而不是一条统一的数据分析链。

### 3.3 live 入口在主流程里权重过高

当前主流程页里仍保留了较多 live / rerun / custom sweep 入口。这会导致：

- 用户容易把主流程误解成“在线调参器”
- 已经冻结的结果库价值没有被突出
- dashboard 的主路径变得更难解释、更难测试、更难维护

## 4. 改造总原则

本轮 dashboard 改造必须同时满足以下原则。

### 4.1 单一事实来源

前 6 页所有关键图表、结论、推荐、切片分析，都应默认来自标准：

- `fine_full_range_biomimetic_exosome_10000e_summary.csv`
- `fine_full_range_biomimetic_exosome_10000e_compact.pkl`
- `fine_full_range_biomimetic_exosome_10000e_meta.json`
- `fine_full_range_biomimetic_exosome_10000e_result_health.json`
- `fine_full_range_biomimetic_exosome_10000e_freeze_probe.json`

### 4.2 主流程只做“结果库分析”

前 6 页的任务是：

- 解释标准结果库
- 切片标准结果库
- 选择标准结果库中的 case
- 分析标准结果库里不同区域的规律

而不是：

- 现场重跑 sweep
- 随手改一组连续参数重新算
- 把每一页都做成半个计算器

### 4.3 自定义计算只放在最后一页

只有 `Single-Case Calculator` 保留自定义输入与单次计算能力。  
这页应明确是**流程外工具页**，不是前 6 页的并行实现。

### 4.4 UI 层隐藏 coarse

`coarse` 可以保留为工程内部资产，但在主 UI 中：

- 不作为默认结果源
- 不作为前 6 页的主切换入口
- 不参与主叙事

### 4.5 先收口，再扩展

本轮先解决：

- 默认数据源唯一化
- 页面职责边界唯一化
- 结果库分析逻辑唯一化

然后才考虑更进一步的视觉优化或机制页深化。

## 5. 目标架构

### 5.1 页面分层

改造完成后的 dashboard 应分成两层：

#### A. 标准结果库分析层

包含前 6 页：

1. `Principle Guide`
2. `Mie Explorer`
3. `Interference Explorer`
4. `Noise & Detection Explorer`
5. `Design Explorer`
6. `Case Inspector`

这一层统一服务于：

- 标准 `fine_full_range_*` 结果库
- 结果解读
- 规律分析
- 候选筛选
- case 解释

#### B. 独立计算工具层

包含最后 1 页：

7. `Single-Case Calculator`

这一层统一服务于：

- 用户自定义输入
- 单次 live 计算
- 完整计算链剖面

### 5.2 状态分层

应明确区分两种状态：

#### A. 数据库分析态

适用于前 6 页，核心状态只应包括：

- 当前标准数据前缀
- 当前选中的粒子类型 / 波长 / W / H / case
- 当前筛选条件
- 当前分析指标

#### B. 单案例计算态

只适用于最后 1 页，核心状态只应包括：

- 单案例输入参数
- 单案例计算结果
- 单案例阶段性报告

这两套状态不应再互相污染。

### 5.3 后端分层

应把 dashboard 后端清晰分为三类能力：

#### A. 结果库读取层

负责：

- 加载 `summary / compact / meta / result_health / freeze_probe / gate_calibration`
- 构造标准 dataset bundle
- 提供统一数据源说明

#### B. 结果库分析层

负责：

- 聚合统计
- 分组切片
- case 定位
- recommendation / gate / freeze 的解释整合

#### C. 单案例计算层

负责：

- 当前输入参数下的单次 live 计算
- 阶段性计算结果组织
- 单案例图表与结论支撑

## 6. 页面职责重定义

### 6.1 `Principle Guide`

定位：

- 整个 dashboard 的读图指南与数据口径说明页

它应明确告诉用户：

- 前 6 页都在分析标准 `fine_full_range_*`
- recommendation / gate / freeze 分别表示什么
- 何时应该去 `Single-Case Calculator`

### 6.2 `Mie Explorer`

定位：

- 标准结果库里的本征散射解释页

它不应再只是一个独立的自由计算页，而应优先回答：

- 在全量结果库中，不同粒子 / 粒径 / 波长的本征散射结构如何变化
- 当前选中 case 的 Mie 位置处于全库的什么区间

### 6.3 `Interference Explorer`

定位：

- 标准结果库里的干涉形成机制解释页

它应优先展示：

- 选中 case 在标准结果库中的 `reference / scattering / overlap / phase` 关系
- 这些机制量在同类 case 中是否异常、是否有优势、是否受 freeze 约束

不应再让用户在这一页自由改连续参数后重算。

### 6.4 `Noise & Detection Explorer`

定位：

- 标准结果库里的读出与检出解释页

它应优先展示：

- `stable_detection_rate`
- `phase_flip_fraction`
- `local_snr`
- `threshold / readout / bandwidth` 的解释关系

同样不应把这页继续当成 live 调参页。

### 6.5 `Design Explorer`

定位：

- 标准结果库的总入口与候选筛选中心

它应承担：

- 全局筛选
- 排序
- 热图
- 分组比较
- 推荐候选入口

这一页应成为前 6 页里最明显的“全库分析中心”。

### 6.6 `Case Inspector`

定位：

- 标准结果库中单个 case 的深度解释页

它应承担：

- 展示选中 case 的关键字段
- 展示批量统计
- 展示单事件样本
- 汇总 recommendation / gate / freeze / blocker

它应该是前 6 页主流程的最后一站。

### 6.7 `Single-Case Calculator`

定位：

- 流程外的独立工具页

它只做一件事：

- 对用户手填参数进行单次完整计算，并逐阶段解释

它不负责：

- 替代标准结果库分析
- 替代 Design Explorer 排序
- 承担 dashboard 主叙事

## 7. 数据源策略

### 7.1 默认数据源

主流程默认数据源必须切换为：

- `fine_full_range`

这意味着：

- `dashboard/app.py` 的默认前缀逻辑应首先指向 `fine_full_range_*`
- 前 6 页默认都基于这一前缀加载 bundle

### 7.2 coarse 的处理原则

`coarse` 结果只保留给：

- smoke test
- benchmark
- precompute 流程验证
- 回归验证

不再作为 dashboard 的主展示入口。

### 7.3 live sweep 的处理原则

`live sweep` 不再属于前 6 页的正常使用路径。  
可以保留底层能力，但应满足：

- 不在前 6 页默认暴露
- 不参与主结论
- 不改变主流程对标准结果库的单一叙事

### 7.4 single-case live 的处理原则

`run_single_case_live(...)` 只服务：

- `Single-Case Calculator`

不再作为前 6 页的隐藏并行路线。

## 8. 代码重构路线

### 阶段 A：先把主流程的数据源收口

目标：

- 前 6 页统一默认使用 `fine_full_range_*`
- 前 6 页移除或明显降级 `coarse / live` 分支

主要改动：

- 调整 `dashboard/app.py` 的默认 prefix 选择逻辑
- 调整 `dashboard/backend.py` 的 bundle 加载入口
- 调整 sidebar / session 的数据源说明文案

完成标志：

- 用户首次打开 dashboard 时，主流程默认就是标准 `fine_full_range` 结果库
- 前 6 页不再需要用户先理解 coarse/live 差异

### 阶段 B：重定义前 6 页职责

目标：

- 把前 6 页全部收成“结果库分析页”

主要改动：

- `Principle Guide`：重写口径说明
- `Mie Explorer`：改成结果库中的 Mie 解释页
- `Interference Explorer`：改成结果库中的干涉解释页
- `Noise & Detection Explorer`：改成结果库中的检出解释页
- `Design Explorer`：强化为全库分析中心
- `Case Inspector`：强化为单 case 深度解释页

完成标志：

- 前 6 页都可以在不触发 live 重算的前提下完成主要阅读路径
- 用户可以只靠标准结果库完成“看规律 -> 找候选 -> 看成因”的闭环

### 阶段 C：隔离单案例计算

目标：

- 把自定义计算明确收敛到最后一页

主要改动：

- 明确 `Single-Case Calculator` 的工具页定位
- 清理前 6 页里残留的单案例 live 入口
- 保证单案例页与主流程页状态隔离

完成标志：

- 主流程与工具页边界清晰
- 用户不会再混淆“结果库分析”和“现场计算”

### 阶段 D：统一解释语言

目标：

- recommendation / gate / freeze / blocker / caution 的解释在前 6 页保持一致

主要改动：

- 提炼统一 caption / badge / status label / guidance 模板
- 减少每一页各自写一套解释文案

完成标志：

- 同一个 case 在不同页面看到的核心解释不互相打架

### 阶段 E：清理冗余实现

目标：

- 减少 dashboard 里重复的数据源切换、session key、状态说明、结果解释模板

主要改动：

- 收紧 `backend / common / panels` 间的共享 helper
- 移除前 6 页中已经失去产品意义的旧分支
- 同步压缩说明文档中的重复叙述

完成标志：

- 代码分支更少
- 文档口径更单一
- 测试更容易锁定主路径

## 9. 推荐实施顺序

建议严格按下面顺序推进，而不是并行大改：

1. **先做阶段 A**
   - 先把默认数据源和主流程叙事收口

2. **再做阶段 B**
   - 先从 `Design Explorer` 和 `Case Inspector` 开始
   - 再改 `Interference Explorer` 和 `Noise & Detection Explorer`
   - 最后再处理 `Mie Explorer`

3. **再做阶段 C**
   - 单独整理 `Single-Case Calculator` 的定位与状态边界

4. **最后做阶段 D / E**
   - 收口统一解释语言
   - 清理残余冗余实现

## 11. 验收标准

### 11.1 产品层验收

- 用户首次打开 dashboard，不需要理解 `coarse / live / fine` 差异

### 11.3 数据层验收

- 前 6 页的关键图表和结论都能追溯到标准 `fine_full_range_*`
- 页面之间对同一 case 的核心判断一致
- `result_health / freeze_probe / engineering_gate_calibration` 被真正用作解释依据，而不是只作为附件文件存在

### 11.4 文档层验收

- `README.md`

- `19_dashboard_explorer.md`
