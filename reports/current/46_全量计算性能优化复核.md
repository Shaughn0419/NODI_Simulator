# 46. 全量计算性能优化复核

更新日期：2026-05-02

2026-05-02 补充状态：本文的性能结论仍成立；本轮只同步 full-grid preflight、selected-annulus paired-valid comparison guard、P0.5 annulus sensitivity 签发结论，以及 Phase 2 target-audit 边界，不改变正式 precompute 的 scalar/off 执行口径。

## 目的

在不改变计算结果的前提下，复核当前全量计算链条是否还有可安全优化的空间，并确认已实施优化不会破坏主链结果。

## 本轮已实施的无结果改动优化

### 1. summary-only 的 `in_phase` 读出走轻量路径

当前 sweep / precompute 正式计算使用 `retain_event_traces=False`。在这个路径下，如果 `readout_observable_mode=="in_phase"`，读出层不再额外计算：

- `signal_nodi_q`
- `signal_nodi_mag`
- `signal_pod_q`
- `signal_pod_mag`
- `signal_nodi_leak`
- `signal_pod_leak`

而是只保留 batch summary 真正会用到的 `signal_detect / signal_nodi / signal_pod / signal_*_true` 和 transit diagnostics。

这个优化不改变 `signal_detect`、threshold、pulse 判定和 summary 指标，只去掉 summary 不消费的诊断数组计算。

### 2. summary-only 的 pulse extraction 跳过 area/prominence

对于 sweep / precompute 的 summary-only 路径，`extract_pulse_features()` 不再为每个峰额外计算：

- `prominence`
- `peak_area`

原因是当前 batch summary / engineering gate / dashboard 主链只消费：

- `peak_time_s`
- `peak_height`
- `peak_signed_height`
- `peak_width_s`
- `peak_polarity`

因此这部分属于纯计算开销，不影响当前正式结果。

### 3. summary-only event 提前返回紧凑结果

`simulate_one_event()` 在 `retain_full_payload=False` 时，不再构建完整 event payload 后再裁剪，而是直接返回 summary 所需的紧凑事件字典。

这一步减少了：

- 大数组挂载
- 大字典构建
- 无用字段复制

但保留了 `summarize_batch()` 当前需要的全部字段。

### 4. artifact profile 减少最终导出 I/O

`dashboard.precompute` 现在支持：

- `--artifact-profile full`
- `--artifact-profile standard`
- `--artifact-profile minimal`

默认 `standard` 已成为正式全量重算导出口径。`standard` 保留 dashboard 必需的：

- `summary.csv`
- `compact.pkl`
- `meta.json`
- `result_health.json`
- `runtime_performance.json`
- 可选 `freeze_probe.json`
- 设计复盘用 `design_postprocess.csv`

同时跳过重复的 `case_summary.csv` 和 heavy parquet split exports：`case_summary.parquet / physics_fields.parquet / diagnostics_long.parquet`。

8-worker quick smoke 中，`standard` 与 `full` 的 summary 按物理 case 排序后完全一致；产物体积从约 `21.18 MiB / 10 files` 降到约 `12.71 MiB / 6 files`。

### 5. TrajectoryContext 复用 case-level 轨迹常量

`nodi_simulator/trajectory.py` 新增 `TrajectoryContext` 和 `build_trajectory_context()`。`run_single_case_batch()` 会为每个 case 构建一次 context，并把它传给每个 event 的 `simulate_particle_trajectory()`。

复用内容包括：

- `n_samples`
- `dt_s`
- `time_s`
- 粒子中心可达的 `half_w / half_h`
- `rect_series` 的 mean factor 与 odd-series 系数

它不缓存随机轨迹，不改变初始位置采样、扩散随机数或反射边界逻辑。

代表性 1000-event 单 case 对照：

- no-context mean：`1.9881 s`
- context mean：`1.9440 s`
- 提升约 `2.27%`
- summary exact failures：`0`

### 6. worker-local case context cache

`nodi_simulator/parameter_sweep.py` 的 sweep worker 现在在进程内复用三类 case-invariant payload：

- `intrinsic scattering`
- `reference field base`
- `collection operator`

这些 cache 通过 worker initializer 放进每个 worker 的 shared context。它们只在当前 worker 进程内有效，不跨进程共享，不写入结果文件，也不改变 RNG 顺序。collection operator 的构建仍沿用旧路径中从 `intrinsic["k_m"]` 反推 medium refractive index 的浮点路径，避免引入 1e-12 级别的 summary 差异。

代表性重复 reference 场景小基准：

- 8 个 gold 粒子
- 同一 `W/H/λ`
- 每粒子 `120` events
- uncached mean：`2.0193 s`
- cached mean：`1.9938 s`
- 提升约 `1.28%`
- summary mismatches：`0`

### 7. summary-only post-readout 诊断瘦身

summary-only 路径不再 materialize 未消费的 NODI post-readout diagnostic payload。若 `post_readout_noise_std > 0`，仍显式调用 RNG 生成同 shape 的噪声并丢弃，以保持后续 POD 噪声序列不变。

固定 seed 对照和 8-worker quick smoke 均确认 summary 不变。

### 3. 新增的结构优化复核

#### A. summary-only 流式累计

这项优化现在已经真正落地到 sweep worker 路径里：

- `stream_summary_only=True`
- `retain_event_traces=False`

时，不再保留整批 slim `events` 列表，而且 `simulate_one_event()` 会直接把 summary 所需量喂给 accumulator，不再先构造 slim event dict 再回收。

验证结果：

1. `1000 events` 代表 case
   - summary 完全一致
   - `events` 长度从 `1000` 变成 `0`
   - 单 case wall-clock 基本持平，平均约 `+0.25%` / `-0.25%` 量级波动，可视为近似持平

2. `10000 events` 代表 case
   - summary 完全一致
   - `events` 长度从 `10000` 变成 `0`
   - 单 case wall-clock 仍接近持平，波动在 `~0.5%` 量级

3. payload 体积
   - `1000 events` slim event 列表 pickle 体积约 `620,503 bytes`
   - 流式路径的 `events=[]` 只有 `5 bytes`
   - event payload 持有体积下降约 `99.999%`

结论：

- 这项优化对单 case CPU 时间提升不明显
- 但它显著降低了 worker 峰值持有数据量
- 在正式 `28 workers × 10000 events/case` 全量重算里，更重要的价值是减小并行内存压力与对象回收负担

#### B. 并行 worker 共享上下文

## 最终优化边界

### A. 已完成的 event 内部核函数收口

在前一轮判断中，下一优先级是 event 内部数值核函数。2026-04-26 已完成并保留的主线优化包括：

- `event_block_v3`：summary-only 路径按 event block 推进；显式启用时默认块大小仍按 `32` 对照
- 2D light illumination kernel：`compute_illumination_envelope(..., export_full_diagnostics=False)` 在 2D block 输入下复用 Numba kernel
- light in-phase readout kernel：跳过 summary 不消费的 Q / magnitude / leak 诊断数组
- block-level peak summary extraction：直接从 2D signal block 提取 best peak summary
- exact row-wise threshold stats：使用每行 median / MAD 的精确 partition 路径
- plug-diffusive trajectory block：默认 `plug + diffusion + no hindrance` 子路径保持 scalar RNG 顺序

这些优化仍属于“不改变统计口径”的可复用路径，因为默认 RNG 顺序保持为
`event_loop_order`；不过 2026-04-27 之后正式全量默认已经切回 scalar/off。

### B. 当前固定默认

当前正式 precompute 默认口径固定为：

- `vectorized_event_engine = "off"`
- `event_block_size = 32`
- `event_block_rng_order = "event_loop_order"`

2026-04-27 在 16C/32T 正式计算机器上补做 10000-event / 64-case / 28-worker 对照后，正式全量默认从 `event_block_v3_b32` 切回 scalar/off：

- `scalar_off_b32`: `74.062 s`
- `event_block_v3_b32`: `131.048 s`
- `event_block_v3_b16`: `107.604 s`

因此当前默认优先保证全量 wall-clock 与保守结果语义。`event_block_v3` 仍保留为显式性能实验/回归对照路径，不作为正式全量默认。

64-case / 3000-event / 8-worker 对照结果：

- scalar: `61.0797 s`
- `event_block_v3_b32`: `39.1546 s`
- 加速：约 `1.56x`
- detection-rate delta: `0`
- stable-detection-rate delta: `0`
- max peak-height delta: `2.22e-16`
- max peak-width delta: `1.30e-18`
- engineering gate agreement: `1.0`

同一基准中，`block_lane_order` 为 `37.9006 s`，相对默认快约 `3.2%`，但：

- engineering gate agreement: `0.9375`
- max detection-rate delta: `0.031`
- max stable-detection-rate delta: `0.034`
- max peak-height delta: `0.0525`
- max peak-width delta: `1.4512e-4 s`

因此 `block_lane_order` 不进入默认主线，只保留为显式实验选项。

### C. 后续优化边界

当前不建议继续在主线做性能重构。剩余能明显改变 wall-clock 的方向主要是：

- 彻底向量化 event loop
- 改变 RNG 消费顺序
- 更激进地合并光学 / 轨迹 / 读出 kernel

这些方向都会更容易改变个体 event 轨迹或浮点路径。只有当独立实验分支能证明明显速度收益，并且检测率、峰宽、峰高、稳定检测率和最终工程结论的偏差可接受时，才应考虑替换默认。

## 当前判断

1. 这轮已经落地的优化是安全的，可以保留
2. 它们已经被回归和数值对照验证过，没有发现结果偏移
3. `stream_summary_only` 的主要收益是显著降内存持有，不是单 case CPU 加速
4. `worker shared context` 明确减少了 case 提交负担
5. `artifact-profile standard` 是正式全量库推荐导出 profile；只有历史兼容 split exports 需求时才切回 `full`
6. `TrajectoryContext` 与 worker-local invariant cache 是低风险小收益优化，适合保留；它们的价值会随粒子数、共享几何/波长重复度和 worker 数增加而更明显
7. 默认事件引擎固定为 `scalar/off + event_loop_order`
8. `block_lane_order` 和彻底向量化 event loop 目前不进入主线

## 结论

当前计算链条已经固定到可进入正式全量重算的性能口径。已经实施的优化可以保留；主线不建议继续为了小幅速度收益重构。下一步应运行正式 full-grid recompute，并在全量结果生成后更新 dashboard 默认数据说明、设计表和最新选型报告。
